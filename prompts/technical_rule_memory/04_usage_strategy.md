# 使用策略

## 不建议的方式

不要把整张 `问题对应技术规则示例(1).xlsx` 逐行拼进提示词中反复发送给模型。

原因：

- token 消耗高
- 噪音大
- 示例过多反而稀释判断边界
- 重复发送同样内容成本过高

## 推荐方式

### 方式一：长期记忆 + 当前样本

输入给模型：

- 本目录中的概览与核心分类记忆
- 当前待判断的 `item_full_title + item_content`

适用场景：

- 先做稳定分类
- 控制 token 消耗

### 方式二：长期记忆 + 候选分类召回

流程：

1. 用关键词或规则先召回 3-8 个候选分类
2. 再把这些候选分类的记忆块发给模型
3. 模型做最终判断

适用场景：

- 大批量跑数
- 进一步减少 token 成本

### 方式三：长期记忆 + 极少量样例

做法：

- 主体用记忆文档
- 只补充 1-3 条最典型 few-shot 样例

适用场景：

- 模型在某些分类边界上仍不稳定

## 召回优先级建议

逐条判断时，优先先检查：

1. 缺流程要素
2. 流程时间倒置
3. 长期挂账未处理
4. 财务做账不规范
5. 内容一致性问题

如果都不明显，再扩展到：

- 合同内容存在问题
- 业务高度重合
- 违反相关规定
- 关联交易
- 其他问题

## 适合后续程序化落地的方向

这组记忆文档后续可以进一步被拆成：

- 分类名
- 触发词
- 强特征
- 弱特征
- 反例边界

这样就能继续做成：

- 候选分类召回词典
- 规则引擎
- 低 token 的提示模板
